# Accessibility (a11y)

The Nurtured Oven targets **WCAG 2.1 Level AA**. This doc describes what was implemented, how to test, and contrast conventions.

## Implemented patterns

| Area | Location | Pattern |
|------|----------|---------|
| Skip link | [`app/layout.tsx`](../app/layout.tsx), [`app/globals.css`](../app/globals.css) | First focusable element; jumps to `#main-content` |
| Mobile menu | [`components/layout/MobileNav.tsx`](../components/layout/MobileNav.tsx) | Renders only when open; `role="dialog"` + `aria-modal`; focus trap + Escape; focus returns to menu button |
| Focus trap hook | [`hooks/useFocusTrap.ts`](../hooks/useFocusTrap.ts) | Reusable Tab cycle + Escape for modals |
| FAQ accordions | [`components/ui/Accordion.tsx`](../components/ui/Accordion.tsx) | `aria-expanded`, `aria-controls`, `hidden` on collapsed panels (content removed from a11y tree) |
| Order cart status | [`components/order/WeeklyOrderCart.tsx`](../components/order/WeeklyOrderCart.tsx) | `role="status"` + `aria-live="polite"` when cart is empty |
| Form errors | [`components/contact/ContactOrderForm.tsx`](../components/contact/ContactOrderForm.tsx) | `role="alert"` with high-contrast `text-espresso` |
| Focus rings | [`app/globals.css`](../app/globals.css) | `:focus-visible` on links, buttons, inputs, selects, textareas |

## Manual testing checklist

Run these before major releases or visible UI changes.

### Keyboard (desktop)

1. Tab from page load: **Skip to main content** appears and activates correctly.
2. Tab through header nav without entering hidden mobile menu links (resize to desktop width).
3. At mobile width: open menu → Tab stays inside drawer → Escape closes → focus returns to hamburger button.
4. FAQ page: expand/collapse items; collapsed answers are not read when tabbing past buttons.
5. Contact `/contact`: switch intent tabs with arrow keys; all fields reachable; submit with empty cart shows alert.
6. Home gallery lightbox (if open): Escape closes dialog.

### Screen reader (VoiceOver on Mac or NVDA on Windows)

Sample pages: `/`, `/contact`, `/faq`, `/menu`.

- Page title and single `<main>` landmark announced.
- Mobile menu announced as dialog when open; closed menu links absent from rotor/link list.
- FAQ: only expanded answer text is available in reading order.
- Form fields: label + required state announced; empty-cart status announced when last item removed.

### Automated checks

```bash
pnpm dev
```

Then in Chrome DevTools:

1. **Lighthouse** → Accessibility on `/`, `/contact`, `/faq`.
2. **axe DevTools** extension on the same pages.

Target: no critical/serious violations on primary customer flows.

## Color contrast conventions

Brand palette (see [`tailwind.config.ts`](../tailwind.config.ts)):

- **Body text:** `text-espresso` on `bg-cream`.
- **Secondary copy on light backgrounds:** use component classes from [`app/globals.css`](../app/globals.css): `text-muted`, `text-muted-sm`, `text-eyebrow`, `text-caption`, `text-hint` (all based on `text-espresso` opacity, not `text-brown-sugar/*`).
- **Avoid** `text-brown-sugar/50`–`/70` for paragraphs; olive-toned brown-sugar fails WCAG on cream.
- **Form labels:** `text-espresso`.
- **Errors:** `text-espresso font-medium` with `role="alert"`.
- **Text on dark hero/footer:** `text-cream/85` or higher for body; `/95` on photo overlays.
- **Placeholders:** `placeholder:text-espresso/45`; labels carry meaning.

When adding new copy, run the DevTools contrast checker (4.5:1 normal text, 3:1 large text).
